Incident Summary

What Happened

The attacker never touched a single line of Axios source code. Instead, they injected a malicious package one level deeper in the dependency tree — invisible to almost every standard security review.

~18 hours before the attack

🎭 Stage the Trojan Package

Attacker published plain-crypto-js@4.2.0 — a completely clean package — to establish registry history and avoid "new package" security alerts.

Prior to March 31, 2026

🔑 Compromise Maintainer Account

A long-lived classic npm token for jasonsaayman (primary Axios maintainer) was obtained. The registered email on the account was changed to ifstap@proton.me, locking out the real maintainer.

March 30, 2026 · 23:59 UTC

🐍 Publish Malicious Transitive Dep

plain-crypto-js@4.2.1 published with a postinstall hook pointing to setup.js — a cross-platform RAT dropper. A clean package.md was bundled inside for the manifest-swap trick.

March 31, 2026 · 00:21–01:00 UTC

☠️ Poison Axios

Using the stolen token, axios@1.14.1 (00:21 UTC) and axios@0.30.4 (01:00 UTC) were published. Both added plain-crypto-js@4.2.1 as a runtime dependency. Zero Axios source lines were modified.

Upon victim's npm install

💀 RAT Deployment & Cleanup

npm automatically executed the postinstall hook, dropping a platform-specific RAT. After launch, the dropper overwrote package.json with the clean manifest and deleted itself — leaving no forensic trace.

Technical Deep Dive

Attack Flow

Eight phases (0–7) from credential theft to silent self-deletion.

0

Staging — Clean Package First

plain-crypto-js@4.2.0 published clean to establish npm history. Common technique to avoid new-package security scanner alerts.

1

Account Compromise via Classic npm Token

Classic npm tokens don't expire and aren't scoped — a stolen token grants full publish access to every package owned by that account forever.

2

Payload Publication

plain-crypto-js@4.2.1 adds "postinstall": "node setup.js" to its package.json. The bundled package.md (a JSON manifest intentionally disguised with a .md extension) is the pre-staged clean manifest used in Step 7.

3

Axios Poisoning (39-Minute Window)

Two poisoned Axios versions inject plain-crypto-js as a runtime dep. No Axios source was modified. The rest of the package was byte-for-byte identical to legitimate prior versions.

4

Victim Installation — Automatic Hook Execution

npm install axios → npm resolves plain-crypto-js as a transitive dep → postinstall fires automatically before any developer code runs. No user interaction required.

5

OS Detection & Payload Delivery

setup.js fingerprints the OS and POSTs to the C2 endpoint. The server responds with the platform-appropriate second-stage payload.

PlatformC2 EndpointDelivery Method
🍎 macOS/product0AppleScript → C++ RAT binary
🪟 Windows/product1VBScript → PowerShell RAT
🐧 Linux/product2Python RAT (/tmp/ld.py)
6

RAT Execution — Remote Access

RATs beacon every 60 seconds. Supported C2 commands: run_payload, shell, ls, kill. No persistence — the RAT dies on reboot, suggesting rapid credential theft was the primary goal.

7

Forensic Self-Deletion

After launching the RAT, setup.js reads the pre-bundled package.md (a valid JSON manifest disguised with a .md extension to appear innocuous), overwrites package.json with it (removing the postinstall hook), then deletes itself. Post-infection inspection of node_modules/plain-crypto-js shows a perfectly clean package.

Why This Attack Was Hard to Catch

🕵️

Transitive Dependency

Most developers only audit direct dependencies. plain-crypto-js was a dep-of-a-dep — invisible to grep-based auditing.

📦

Never Imported

No require('plain-crypto-js') anywhere in the Axios source. Pure postinstall side-effect — fully invisible to static analysis.

🧹

Self-Deletion

The dropper removed all evidence after executing. Post-incident forensics on the node_modules folder found nothing suspicious.

🔑

Classic Token Abuse

Long-lived classic npm tokens leave no OAuth trace and never expire. Once stolen, they grant permanent publish access.

Quick Detection Checklist

# 1 — Check Axios version
npm list axios
cat node_modules/axios/package.json | grep '"version"'
# If you see 1.14.1 or 0.30.4 → AFFECTED

# 2 — Check for plain-crypto-js
ls node_modules/plain-crypto-js 2>/dev/null && echo "FOUND — investigate"

# 3 — macOS: check for RAT binary
ls -la /Library/Caches/com.apple.act.mond 2>/dev/null && echo "RAT FOUND"

# 3 — Linux: check for Python RAT
ls -la /tmp/ld.py 2>/dev/null && echo "RAT FOUND"

# 4 — Check for C2 network connections
lsof -i | grep sfrclak
netstat -an | grep sfrclak

# 5 — Block the C2 domain immediately
echo "0.0.0.0 sfrclak.com" | sudo tee -a /etc/hosts

Preventive Controls

How to Prevent This

Multiple controls at different layers would have caught or blocked this attack.

🔒

Scoped & Time-Limited npm Tokens

Use granular tokens scoped to specific packages with short TTLs. Classic tokens never expire and aren't scoped.

🔑

TOTP / Passkey on npm Accounts

Require MFA on publish. Even a stolen token can't be used without the second factor.

🚫

Disable postinstall Scripts

Add ignore-scripts=true to .npmrc or use npm install --ignore-scripts in CI. Eliminates this entire attack class.

📋

Lockfile Pinning & Diffs

Review package-lock.json diffs in every PR. The addition of plain-crypto-js would have been immediately visible.

🔍

Socket / StepSecurity / Snyk

Automated tools that flag postinstall scripts in new or modified transitive deps as high-risk — exactly what this attack used.

✅

npm Provenance Attestation

Cryptographically verifies that a package was built by the expected GitHub Actions CI. The poisoned versions had no provenance.

🌐

Egress Firewall in CI/CD

Block unexpected outbound connections during npm install. The dropper's POST to the C2 would have been blocked.

🤖

Renovate / Dependabot PRs

Surfaces dependency changes as reviewable PRs before they hit production. Transitive dep additions become visible to the team.

Recommended: Disable Scripts in CI

# Add to your .npmrc
ignore-scripts=true

# Or per-install
npm install --ignore-scripts

Note: This may break packages that legitimately use postinstall (e.g. esbuild, node-sass). Evaluate on a per-project basis.
